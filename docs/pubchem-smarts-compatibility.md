# PubChem SMARTS compatibility

PubChem documents SMARTS input for substructure search and says it recognizes standard SMARTS extensions, including recursive SMARTS:

- https://pubchem.ncbi.nlm.nih.gov/search/help_search.html
- https://daylight.com/dayhtml/doc/theory/theory.smarts.html

PubChem does not publish a complete parser grammar, so the compatibility profile is based on the documented support plus direct PUG-REST probes against constructs generated by this project. It is intentionally stricter than "does PubChem return any response": some constructs are accepted or silently degraded by PubChem but rejected by other PubChem-oriented SMARTS tools such as smarts.plus. CDK's SMARTS flavor documentation is useful background for identifying which constructs are toolkit extensions rather than core Daylight syntax:

- https://cdk.github.io/cdk/2.7/docs/api/org/openscience/cdk/smarts/Smarts.html

`EvolutionConfig::builder().pubchem_compatible_smarts(true)` enables the profile at runtime. Full SMARTS remains available when that method is left disabled, while the web worker enables PubChem-compatible generation by default through its request configuration.

## Allowed

The profile allows the core SMARTS constructs used by the GA and accepted by PubChem probes:

- bare and bracket atoms, wildcard atoms, atomic number, and portable element symbols
- exact numeric atom predicates such as `D2`, `X3`, `v4`, `H1`, `R`, `R2`, `r6`, and `x2`
- atomic numbers from `#1` through `#118`
- boolean atom and bond expressions using `!`, `&`, `,`, and `;`, without nested negation
- recursive SMARTS such as `[$([#6]=[#8])]`
- atom maps up to `999`, such as `[#6:1]`
- component grouping and dot-disconnected fragments
- formal charge, normal isotope mass labels, and common chirality classes `@`, `@@`, `@TH`, and `@SP`
- single, double, triple, aromatic, any, ring, directional, and elided bonds

## Suppressed

The profile disables mutations that introduce constructs observed to fail PubChem PUG-REST, return unsafe PubChem behavior, or fail PubChem-oriented validation through smarts.plus:

- numeric range predicates, for example `[D{2-4}]`
- extension atom predicates that PubChem can silently ignore, including hybridization `^`, hetero-neighbor count `z`, and aliphatic hetero-neighbor count `Z`
- bare valence `v`; exact valence predicates such as `v2` remain allowed
- implicit lowercase hydrogen count `h`, for example `[c;h1]`
- element-symbol forms rejected by smarts.plus while atomic-number forms work, for example `[Rf]`, `[Sg]`, `[Rg]`, `[Fl]`, `[Mc]`, `[Lv]`, `[Ts]`, and `[Og]`
- uncommon stereochemistry classes `@TB` and `@OH`
- atom maps above `999`, for example `[#6:1000]`
- atomic numbers outside `#1` through `#118`, for example `[#0]` and `[#119]`
- disconnected/component-grouped recursive SMARTS, for example `[$([#6]-[#8].[#7])]`
- nested atom or bond negation, for example `[!!#6]` and `[#6]!!#[#7]`
- wildcard atoms inside negated atom subtrees, for example `[!*]`
- serialized bracket expressions that start with a bare `-` charge primitive, for example `[-]`, `[-&N]`, and `[-:1]`; PubChem accepts equivalent charged atom forms such as `[N-]` and `[N&-]`
- serialized bracket expressions that start with chirality, for example `[@]`, `[@&#6]`, and `[!@]`; PubChem accepts equivalent atom-qualified forms such as `[#6@]` and `[C@H](F)O`
- allene chirality `@AL`; PubChem accepts compact literals such as `[C@AL2]`, but the GA emits canonical SMARTS and `smarts-rs` currently serializes these into PubChem-failing boolean forms
- quadruple bonds, for example `[#6]$[#6]`
- isotope wildcard mass numbers above `255`, for example `[65535*]`

Mutation and crossover also run generated children through a compatibility check when the profile is active, so existing incompatible seed SMARTS are not preserved into accepted offspring unless a mutation removes the incompatible construct.
